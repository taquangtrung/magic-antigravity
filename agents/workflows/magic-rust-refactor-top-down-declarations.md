# Magic Rust Refactor Top-Down Workflow

This workflow guides the agent to systematically restructure Rust modules using a clean, top-down approach. It ensures that the file is logically organized, making it significantly easier for developers to navigate and maintain.

## 1. Goal

Restructure a target Rust module to improve readability and maintainability by declaring all data structures at the top, separating and grouping their implementations below, and cleaning up clutter.

## 2. Process

### Step 1: Analyze and Clean Comments

- Read and understand the target module completely.
- Remove redundant, noisy, legacy, or autogenerated comments that do not add value or clarify complex logic.
- Keep meaningful docstrings (`///` or `//!`) intact.

### Step 2: Declare Data Structures (Top-Down Approach)

- Move all data structure definitions (`struct`, `enum`, `type` aliases) to the top of the module beneath a single group banner.
  ```rust
  // ========================================================================
  // Data Structures
  // ========================================================================
  ```
- Order definitions logically using a top-down hierarchy: if `struct`/`enum` A uses `struct`/`enum` B in its definition, then A is at a higher hierarchy and must be defined first. Therefore, the highest-level structures appear at the top, followed progressively by their supporting or auxiliary lower-level types. Always sort the fields of a `struct` and elements of an `enum` alphabetically by name.
  Example:
  ```rust
  pub struct A {
      pub b: B,
  }

  pub struct B {
      pub id: usize,
  }
  ```

- Any `trait` definitions must be placed in their own separate group with a distinct banner.
- Sort multiple trait definitions alphabetically by the trait's name.
  ```rust
  // ========================================================================
  // Traits
  // ========================================================================
  ```
- Do not interleave `impl` blocks among these definitions.

### Step 3: Group and Separate Implementations

- Move all `impl` blocks (both intrinsic and trait implementations) below the block of data structure definitions.
- Group the implementations by the data structure they implement.
- Sort these implementation groups alphabetically by the data structure's name.
- Separate implementations of different data structures using a distinct comment banner for visual clarity.
  Example block:
  ```rust
  // ========================================================================
  // AssignStmt Implementations
  // ========================================================================

  impl AssignStmt { ... }

  // ========================================================================
  // Block Implementations
  // ========================================================================

  impl Block { ... }
  ```

### Step 4: Verification

- // turbo-all
- After restructuring, verify that the code still compiles successfully.
- Run the cargo check command: `cargo check`
- Wait for the verification to pass. Fix any visibility, ordering, or borrowing errors caused by the code movement before finalizing the task.

## 3. Constraints

- **Zero Functional Changes:** Do not modify the existing logic or behavior of the code.
- **Tests at the bottom:** Any `#[cfg(test)]` modules must remain at the very bottom of the file.
